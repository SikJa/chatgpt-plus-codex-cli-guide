¿Qué es Codex CLI?

Codex CLI es una herramienta de línea de comandos que usa modelos de IA para generar, modificar y depurar código desde tu terminal, dentro de tu repositorio. Propone cambios, instala dependencias y puede ejecutar pruebas con aprobación del usuario. Funciona en macOS/Windows (vía WSL) y Linux. La idea clave: IA + estructura + aprobación humana.

¿Por qué usarlo?

  • Velocidad en scaffolding de proyectos
  • Refactors guiados por IA
  • Explicación automática de código
  • Pasos aprobados por el usuario

¿Qué hace Codex CLI?

Ejemplos prácticos de lo que puedes lograr con Codex CLI

Generar código desde cero

Agregar tests automáticamente

Genera tests completos para tu código:

ejemplo

Corregir bugs y errores

Identifica y soluciona problemas en tu código:

ejemplo 
codex "arreglar los errores de ESLint en src/"

Quickstart

Configura Codex CLI en 4 pasos simples

Prerrequisitos

Asegúrate de tener instalado:

  • macOS, Windows (WSL2), o Linux
  • Node.js 22+ instalado
  • Git configurado

Instalar Codex CLI

Instala globalmente via npm:

bash 
npm install -g @openai/codex

Iniciar Codex

Al ejecutar, te pedirá iniciar sesión con ChatGPT Plus:

bash 
codex

Instalación por Sistema Operativo

macOS

Instala las dependencias básicas. Homebrew es opcional pero recomendado.

bash 
node -v
git --version
npm install -g @openai/codex
codex --help

Windows (WSL2 Ubuntu)

Asegúrate de tener WSL2 habilitado. Dentro de Ubuntu, instala Node 22+ y Git.

bash 
node -v
git --version
npm install -g @openai/codex
codex --help

Linux (Debian/Ubuntu)

Actualiza el sistema e instala las dependencias básicas, Node 22+ y Git.

bash 
sudo apt update && sudo apt upgrade -y
node -v
git --version
npm install -g @openai/codex
codex --help

Comandos esenciales

MUY IMPORTANTE: Control de versiones

Siempre usa Git antes de ejecutar Codex CLI. Asi podes revertir cualquier cambio si algo sale mal.

Antes de usar Codex

bash 
git init\ngit add .\ngit commit -m "Backup antes de Codex"

Si algo se rompe

bash 
git status\ngit diff\ngit checkout -- archivo\ngit reset --hard HEAD

Tip: Codex CLI funciona mejor cuando detecta un repo Git inicializado. Usa commits chicos y frecuentes.

Lanzar TUI

Interface de usuario en terminal:

bash 
codex

Prompt directo

Ejecutar comandos directamente:

bash 
codex "arreglá los errores de lint en src/"

Modo no interactivo

Ejecutar sin confirmación:

bash 
codex exec "explicá utils.ts"

Especificar modelo

Usar un modelo específico:

bash 
codex -m o4-mini "creá un endpoint Express"

Con aprobación

Requerir confirmación explícita:

bash 
codex --ask-for-approval "refactorizá este módulo"

Actualizar Codex

Mantener la herramienta actualizada:

bash 
codex update

Control completo

Dar acceso completo sin confirmaciones:

bash 
codex --approve-all --full-access

# PROMPT: Estructurador Profesional de Proyectos de Software

Actua como un Senior Developer y Tech Lead pragmatico con mas de 10 anos de experiencia. Tu objetivo es ayudarme a planificar, estructurar y organizar un proyecto de software desde cero, maximizando productividad y minimizando friccion.

## FASE 1: DISCOVERY (PREGUNTAS OBLIGATORIAS)

Haz exactamente estas 5 preguntas, una por vez, esperando mi respuesta antes de continuar:

1. PRODUCTO: Que quieres construir exactamente? Describe el problema que resuelve y para quien.
2. ALCANCE: Cual es tu MVP y que debe poder hacer el usuario final?
3. PLATAFORMA: Donde debe correr? (web app, API REST, CLI, mobile app, desktop, etc.)
4. STACK: Tienes preferencias tecnicas o restricciones? (lenguaje, framework, base de datos, hosting, presupuesto)
5. CONTEXTO: Cual es tu nivel tecnico y cuanto tiempo tienes para desarrollar?

IMPORTANTE: Si no respondo alguna pregunta, asume la opcion mas estandar y marcala como [ASUNCION] con una breve justificacion.

---

## FASE 2: ENTREGABLE COMPLETO

Una vez que tengas mis respuestas, entrega lo siguiente en este orden:

### 1. Executive Summary
- Problema: que resuelve
- Usuario: para quien
- Solucion: como lo resuelve
- MVP: funcionalidad minima
- Valor: por que es importante

### 2. Arquitectura del proyecto
```
/proyecto-nombre/
|-- src/                 # Codigo fuente principal
|-- tests/               # Tests automatizados
|-- docs/                # Documentacion
|-- config/              # Configuraciones
|-- scripts/             # Scripts de automatizacion
|-- assets/              # Recursos estaticos
`-- README.md            # Guia del proyecto
```
Detalla:
- Estructura de carpetas especifica
- Responsabilidad de cada modulo
- Flujo de datos entre componentes
- Patrones de diseno recomendados

### 3. Stack tecnico completo
- Runtime: lenguaje y version
- Framework: principal + justificacion
- Base de datos: tipo + motor
- Testing: framework de tests
- DevOps: CI/CD y contenedores
- Tooling: linters, formatters y tasks
- Dependencias clave: top 5-8 librerias

### 4. Roadmap de desarrollo (paso a paso)

#### Sprint 1: Setup and foundation
- [ ] Tarea especifica - DoD medible - Test para verificar

#### Sprint 2: Core features
- [ ] Tarea especifica - DoD medible - Test para verificar

#### Sprint 3: Polish and deploy
- [ ] Tarea especifica - DoD medible - Test para verificar

### 5. Riesgos y mitigaciones
1. Riesgo tecnico - Probabilidad - Impacto - Plan de mitigacion
2. Riesgo de tiempo - Probabilidad - Impacto - Plan de mitigacion
3. Riesgo de complejidad - Probabilidad - Impacto - Plan de mitigacion

### 6. Comandos de inicializacion
```bash
# Setup inicial del proyecto
mkdir proyecto-nombre && cd proyecto-nombre
# Agrega aqui los comandos especificos del stack
```

### 7. Prompts tecnicos para IA
Copia y pega segun necesites:

#### Setup inicial
```
"Genera el scaffolding completo para [PROYECTO] usando [STACK] con la arquitectura [ARQUITECTURA]. Incluye configuraciones de linters, formatters y scripts de desarrollo."
```

#### Desarrollo de features
```
"Desarrolla la funcionalidad [FEATURE] para [CONTEXTO] siguiendo el patron [PATRON]. Incluye validaciones, manejo de errores, tests y documentacion."
```

#### Testing
```
"Escribe tests completos para [MODULO] cubriendo casos felices, edge cases y errores. Usa [TEST_FRAMEWORK] con mocks para dependencias externas. Cobertura minima 80%."
```

#### Refactoring
```
"Refactoriza [CODIGO] aplicando principios SOLID y patrones [PATRONES]. Manten compatibilidad con [DEPENDENCIAS]. Explica cada cambio y su justificacion."
```

#### Debugging
```
"Analiza el error [ERROR] en [CONTEXTO]. Proporciona causa raiz, solucion paso a paso y prevencion futura. Incluye logs de debugging."
```

### 8. Checklist de validacion
- [ ] Resuelve el problema especifico
- [ ] El MVP es alcanzable en el tiempo disponible
- [ ] El stack coincide con la experiencia del equipo
- [ ] La arquitectura es escalable
- [ ] Los riesgos estan identificados
- [ ] Hay claridad para empezar ya

Buenas practicas: "Estructura antes que IA"

Objetivo claro

Define inputs, outputs, entidades y flujos antes de generar codigo.

Arquitectura definida

Establece la estructura de carpetas y capas clave antes de usar IA.

Prompt tecnico especifico

Incluye convenciones, dependencias, tests y limites bien claros.

Error comun

Usar IA sin cerrar la arquitectura termina en codigo inconsistente.

Comparativa: Cursor vs Claude Code vs Codex CLI

Codex CLI

Pros:

  • Terminal/TUI nativo
  • Pasos aprobados por usuario
  • Foco específico en CLI
  • Integración con repositorios Git

Contras:

  • Requiere Node 22+
  • Sin interfaz visual
  • Curva de aprendizaje para comandos

Claude Code

Pros:

  • Integración profunda con IDE
  • Contexto grande de archivos
  • Interface visual intuitiva
  • Soporte multi-lenguaje

Contras:

  • Dependiente de Anthropic
  • Requiere suscripción
  • Menor control granular

Cursor

Pros:

  • Editor completo estilo VSCode
  • Refactors y chat integrado
  • Experiencia visual completa
  • Plugins y extensiones

Contras:

  • Migración de editor existente
  • Más pesado en recursos
  • Costo de suscripción
